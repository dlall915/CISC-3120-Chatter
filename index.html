Project 2: More Chatting

Overview

This project is an extension of the Chatter labs we've been playing with in class. Simply put, you're going to add a little bit more sophistication to the Chatter protocol (just a little), and you're going to give the client and server a nice Graphical User Interface. And for extra credit, you can think about adding a bit more sophistication to the Chatter protocol and its implementation.

Foundations

Notice that there's no code in this repository! Once you've caught your breath, remember that you have plenty of code at your disposal: I gave you a sockets-based implementation of the Chatter protocol, and you are working on an RMI-based implementation. You can use either of those as the basis for this project; just copy that code into this project (and then commit; make sure the new files are added to the repository).

Extending the Protocol

Let's add something that slightly relaxes the rigid "taking turns" nature of the Chatter protocol (which isn't that natural, after all). This is not going to be a huge improvement, but it's something. The new rule is this: if the "current chatter" types a chat which ends in three periods ("..."), then instead of the other chatter getting their turn, the current chatter gets to type another chat. This can go on as long as the current chatter wants to keep talking (that is, bogart the mic); when the current chatter types something that doesn't end in three dots, then the other chatter gets their turn (or turns). Note that these three dots should have no spaces between or after them.

Providing a User Interface

Once you have that working on the console, work on providing a GUI for both the client side and server side--ultimately you should be able to run both of these simultaneously. You have a lot of freedom here: you can choose to implement these as applets (in which case you can steal a lot of code from the applet lectures) or as applications (in which case you might want to steal some code from project 1, also).

This is not a complicated application, so neither should your interface be complicated. But before you begin, you should think about what elements the interface should include, and how they should be laid out. You might want to write the layout code first, before you "hook up" the actual protocol. You should also think about how to integrate the interface fully with the protocol. For example, how can you use the interface to indicate to the user whether it's their turn or not?

Also, design your GUI code with this question in mind: what if someone wanted to use your interface for a different chat protocol, or an even-more-modified version of Chatter? Can you design your interface so that it doesn't directly require Chatter clients/servers? This is something of a challenge; feel free to bounce ideas off me.

Doing More

If you've got all that working and you have time and your hands, start thinking about how you might further relax the Chatter protocol so that your users have a more "natural" chatting experience. What will need to change? What additional rules do you need? Can you remove any existing rules?

Implement your revised protocol in new classes (call the protocol something like ChatterXtreme); if you designed your GUI well, you shouldn't need to do much to hook this new protocol into your GUI.
